## Референсы + твои макеты

Твои макеты очень чётко описали взаимодействие:

1. **Главный экран — Календарь**
    - Слева — статичное меню.
    - В центре — месячный календарь.
    - Справа — панель «Следующие события» с табами:
        - **Паки** — список кратких карточек паков + кнопка «Создать новый пак».
        - **Релизы** — список релизов (дата + краткая карточка) + кнопка «Запланировать новый релиз».
    - В ячейках календаря — подписи «релиз + статус».
2. **Полная карточка пака**
    - Все ключевые поля (RU/EN, теги, тип, статус).
    - Блоки файлов: обложка, скрины, дополнительные файлы — с Replace / Download / Delete.
    - Большой чек-лист по процессу создания пака и постов.
3. **Релизы**
    - Всплывающее окно просмотра релиза:
        - Название, дата/время, пак (с переходом к карточке пака), чекбоксы платформ (TG, Boosty, VK, Tumblr), кнопки редактировать/удалить.
    - Всплывающее окно создания/редактирования релиза:
        - Название, дата (с календарём), время, выбор пака из списка, чекбоксы платформ.
    - Логика:
        - клик по пустой дате или по «Запланировать новый релиз» → модал создания;
        - клик по релизу в календаре или в правой панели → модал просмотра/редактирования.

Связки:

- **Pack** ↔ **Release** : один пак может иметь несколько релизов.
- **Release** ↔ **Platforms** : множество платформ через чекбоксы/enum.
- Из релиза можно провалиться в полный пак.

## 1. Главный референс — Pose Calendar

Что я вижу по структуре:

**1) Трёхколоночный layout**

- **Слева** — боковое меню:
    - Pose Calendar (название/логотип)
    - разделы: *Calendar*, *Packs*, *Analytics*
    - снизу иконка настроек.
- **По центру** — основной календарь (Month view).
- **Справа** — контекстная правая панель с деталями релиза.

Это идеально под твой кейс:

лево — навигация между «календарь / паки / аналитика»,

центр — сетка по датам,

право — «карточка выбранного релиза».

---

**2) Верхняя панель над календарём**

- Название: **Pose Calendar**
- Переключение месяца: *December 2025* + стрелки.
- Справа — дропдауны/переключатели:
    - что-то вроде выбора автора/проекта (Thean),
    - переключатель режима вида — *List* (скорее всего List/Calendar).

→ Это задел на:

- выбор набора контента (например, фильтр по автору или типу: только позы, только скрины),
- переключение между календарём и списком событий.

---

**3) Центральный календарь**

- Month view с крупными ячейками дней.
- В ячейках:
    - цветные бейджи с кодами (по факту — релизы/события),
    - на 16-е число — вертикальная карточка релиза (фиолетовая) + под ней голубой бейдж «TG».

Смысл:

- каждый релиз — это **событие в календаре**,
- цветами кодируется тип (Free, TG, ещё что-то),
- внутри дня может быть несколько событий.

---

**4) Правая панель — контекст релиза**

Наверху:

- Блок **Releases** — список релизов (микро-листа) + кнопка «+ Ariese» (типа «добавить релиз»).
- По клику/выбору — снизу показываются детали конкретного релиза.

Карточка релиза:

- Название (1111)
- ID, статус (Free)
- **To-do список:**
    - Create poses
    - Make screenshots
    - Write a description
    - Prepare text for VK and TG
    - Prepare archive
    - Publish to Puteron
    - Post announcement
- Блок **Pose files**:
    - Preview
    - Screenshots
    - Archive
        
        — и напротив каждому Upload/Replace.
        

То есть справа совмещены:

- статусная карточка релиза,
- чеклист по подготовке контента,
- менеджер файлов для этого релиза.

Это очень точно совпадает с твоей задачей:

- привязать **todo-шаги рабочего процесса** к релизу,
- хранить все файлы пака (обложка, скрины, архив, тексты постов) в одном месте.

---

**5) Нижняя часть под календарём**

Там таблица **Releases** с легендой по цветам и полями вроде:

- цветной квадратик (метка),
- название,
- время,
- тип (Free/TG),
- действия (Edit / Delete).

### Вывод по главному референсу

Он даёт нам очень чёткую модель:

- Центральный объект — **Release** (релиз пака).
- У релиза есть:
    - дата/время,
    - тип (Free/TG/Boosty и т.д.),
    - связанный **Pose Pack**,
    - чеклист задач,
    - набор файлов.

UI-паттерн:

- **Календарь** + **контекстная правая панель** + **табличный список** = идеальный «операционный» экран для креатора.

## 2. Остальные референсы

### 2.1. Второй скрин (Jan’ 2020, горизонтальный календарь)

Особенности:

- Более плоский layout без правой панели.
- Вид «полоса дней» (почти как timeline по неделе).
- События списком под каждым днём, много текста.
- Крупная кнопка **Add event** справа.

Полезно как идея для:

- **List / Week view** — когда нужно не квадратики календаря, а ровный список задач/релизов по дням.

---

### 2.2. Третий скрин (CelebFarm)

Особенности:

- Левое тёмное меню, по центру — **month view**, справа — колонка **Upcoming**:
    - Today,
    - Tomorrow,
    - дальше по датам.
- В ячейках календаря — маленькие цветные бейджи (несколько событий в один день).
- Справа — большие карточки со временем, названием, логотипом бренда.

Полезно:

- Паттерн «Календарь + колонка ближайших событий».
    
    Для жены это может быть:
    
    - «Сегодня надо: выложить пост в VK + ТГ»,
    - «Завтра — общий доступ пака X».

---

### 2.3. Четвёртый скрин (Manageko Calendar)

Особенности:

- Прям классический корпоративный календарь:
    - переключатель Day / Week / Month,
    - шкала времени по вертикали,
    - блоки-события растягиваются по времени.
- Фильтр, поиск, кнопка `+ New`.

Тут ценна идея:

- поддерживать **режимы Day/Week/Month**,
- добавить фильтры (по типу релиза, по площадкам, по статусам — Draft/Ready/Posted).

---

### 2.4. Пятый скрин (Website Redesign — Calendar)

Особенности:

- Слева меню, по центру — месяц, справа — панель конкретного дня:
    - в правой колонке задачи на выбранный день (с чекбоксами).

По сути это очень похоже на твой главный референс, только:

- справа фокус не на *одном релизе*, а на *списке задач дня*.

Идея:

- Можно комбинировать:
    - режим «Правая панель = релизы дня»,
    - режим «Правая панель = детали выбранного релиза».

---

## 3. Что общее у всех референсов

1. **Левая навигационная колонка** — стабильный паттерн.
2. **Центр = календарь**, чаще всего Month view.
3. **Правая колонка**:
    - либо список грядущих событий,
    - либо детали текущего дня / события.
4. **Цветовое кодирование событий** (типы, статусы).
5. **Отдельная кнопка для добавления события** (Add event / + New).
6. Часто есть **переключатель Day / Week / Month / List**.

---

## 4. Как это ложится на твой будущий интерфейс

Если собрать всё из референсов + твой домен, получается почти готовый скелет:

- **Sidebar:**
    - Calendar
    - Packs (список паков с фильтрами, без календаря)
    - Posts / Analytics (статистика по релизам, лайкам, платным/бесплатным и т.п.)
- **Главный экран «Calendar»:**
    - Month view (основной)
    - Переключение на Week/List (в будущем)
    - Цвета:
        - Boosty Early access,
        - Free release,
        - TG-анонс,
        - VK-пост и т.п.
- **Правая панель:**
    - Если выбран день → список релизов этого дня (+ кнопка «Создать релиз»).
    - Если выбран релиз → карточка:
        - Название пака
        - Тип/статус
        - Привязанные площадки (VK/TG/Boosty)
        - Checklist по подготовке
        - Файлы (cover, screencaps, archive, текстовые заготовки постов)
        - Быстрые ссылки «Сгенерировать пост для VK/TG».
