# Описание бизнес проекта

Owner: mv_vm
Number: 1

> Календарь релизов + хранилище паков + менеджер постов для соцсетей.
> 

## Цель и бизнес-требования (сводка)

**Кому и зачем**

- Один (пока) креатор контента по Sims 4 (жена).
- Нужно перестать держать всё в голове/табличках и **управлять паками поз + релизами + постами по площадкам** из одного места.

**Ключевые задачи продукта**

1. **Хранилище паков**
    - Создать «Пак поз» с:
        - названием RU / EN;
        - типом (solo / couple / mixed и т.п.);
        - тегами, тематикой;
        - статусом (idea / in progress / ready / released).
    - Прикрепить файлы:
        - обложка;
        - скриншоты/коллажи;
        - архив .package и др. доп. файлы (инструкции, PSD и т.п.).
    - Хранить текстовые блоки:
        - «Вам понадобится» (зависимости: моды, объекты, DLC);
        - «Инструкция»;
        - TOU (часть общая, часть специфичная).
    - Иметь **чек-лист по паку**:
        - сделать позы, протестировать, сделать скрины, обложку, собрать архив, подготовить посты для площадок и т.д.
2. **Календарь релизов**
    - На уровне календаря планировать **события-релизы**:
        - дата + время;
        - к какому паку относится;
        - на какие платформы (TG, VK, Boosty, Tumblr, …);
        - статус (запланирован / готов / опубликован — минимум).
    - Один пак может иметь несколько релизов (ранний доступ на Boosty, общий доступ в TG/VK и т.п.).
    - Календарь Month view (позже Week/List), цветовое кодирование типов/статусов.
    - Быстро смотреть:
        - что выходит сегодня/на этой неделе;
        - когда планируются бесплатные раздачи vs платный контент.
3. **Менеджер постов по площадкам**
    - На основе данных пака хранить **черновики постов**:
        - текст для Boosty (полный), VK, TG, Tumblr;
        - хэштеги;
        - какие картинки использовать (обложка, конкретные скрины).
    - Привязка к релизу/платформе, пометка «пост подготовлен / опубликован».
4. **Рабочие процессы (workflow)**
    - Чек-листы:
        - по паку (креатив+подготовка);
        - опционально по релизу/площадке (например, «запостить в VK», «добавить ссылку на Boosty»).
    - Возможность быстро увидеть, на каком шаге сейчас пак / релиз.
5. **Пользователи и доступ**
    - Пока один пользователь (жена / ты).
    - Простая аунтентификация (логин/пароль); в моделях заложить сущность `User` на будущее.
6. **Тех.требования**
    - Backend: Java + Spring Boot.
    - БД — реляционная (логично PostgreSQL).
    - Хранение файлов — локальная файловая система (с путями в БД), в перспективе можно вынести в S3/аналог.
    - REST API под фронтенд (скорее всего отдельный SPA, но это можно уточнить позже).
