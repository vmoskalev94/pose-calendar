## Референсы + твои макеты

Твои макеты очень чётко описали взаимодействие:

1. **Главный экран — Календарь**
    - Слева — статичное меню.
    - В центре — месячный календарь.
    - Справа — панель «Следующие события» с табами:
        - **Паки** — список кратких карточек паков + кнопка «Создать новый пак».
        - **Релизы** — список релизов (дата + краткая карточка) + кнопка «Запланировать новый релиз».
    - В ячейках календаря — подписи «релиз + статус».
2. **Полная карточка пака**
    - Все ключевые поля (RU/EN, теги, тип, статус).
    - Блоки файлов: обложка, скрины, дополнительные файлы — с Replace / Download / Delete.
    - Большой чек-лист по процессу создания пака и постов.
3. **Релизы**
    - Всплывающее окно просмотра релиза:
        - Название, дата/время, пак (с переходом к карточке пака), чекбоксы платформ (TG, Boosty, VK, Tumblr), кнопки редактировать/удалить.
    - Всплывающее окно создания/редактирования релиза:
        - Название, дата (с календарём), время, выбор пака из списка, чекбоксы платформ.
    - Логика:
        - клик по пустой дате или по «Запланировать новый релиз» → модал создания;
        - клик по релизу в календаре или в правой панели → модал просмотра/редактирования.

Связки:

- **Pack** ↔ **Release** : один пак может иметь несколько релизов.
- **Release** ↔ **Platforms** : множество платформ через чекбоксы/enum.
- Из релиза можно провалиться в полный пак.

## 1. Главный экран «Календарь»

**Лейаут:**

- Слева — статичное боковое меню (лейбл приложения, инфа о пользователе, настройки, логаут).
- Центр — месячный календарь (month view) без лишнего шума.
- Справа — панель **«Следующие события»** с табами:
    - **Паки**
    - **Релизы**

### Табы в правой панели

**Вкладка «Паки»**

- Список кратких карточек паков:
    - заголовок = название пака;
    - внутри — «карточка карточка пака» (краткий превью-контент).
- Внизу — кнопка **«Создать новый пак»**.
- Стикер: по умолчанию панель показывает именно паки; по клику по краткой карточке открывается **полная карточка пака** (отдельный большой экран).

→ Логика:

- Пак — это основная единица контента.
- У него есть **краткая карточка** (для списков) и **полная карточка** (редактирование).

**Вкладка «Релизы»**

- Список карточек релизов:
    - сверху — дата (`Дата ...`),
    - под ней — короткое описание события релиза конкретного дня.
- Внизу — кнопка **«Запланировать новый релиз»**.
- Стикер: при переключении на «Релизы» заголовок = дата релиза, и показывается краткая карточка события.

→ Логика:

- Релиз = событие во времени, привязанное к паку.
- Релиз всегда «живет» внутри календаря.

## 2. Полная карточка пака

Отдельный экран (скорее всего на весь центр/правую часть):

Поля:

1. Название пака на RU
2. Название на ENG
3. Дата события релиза (если есть) — скорее информационно, основная логика по датам в сущности «релиз».
4. Тип (например, solo/couple, free/paid и т.п.)
5. Теги
6. Статус (черновик, в работе, готов, опубликован и т.п.)
7. Файл обложки + превью + кнопки **заменить / скачать / удалить**
8. Файлы скринов + превью + **заменить / скачать / удалить**
9. Дополнительные файлы (например, инструкции, TOU, PSD обложек) + **заменить / скачать / удалить**
10. Чек-лист:
    - Сделать позы
    - Протестировать в игре
    - Сделать скрины
    - Сделать обложку
    - Подготовить текст поста
    - Собрать .package / архив
    - Подготовить пост для Telegram
    - … для VK
    - … для Boosty
    - … для Tumblr

Плюс где-то на карточке — кнопки **закрыть / удалить**.

→ Это по сути **центр управления паком**:

- все файлы,
- вся мета-информация,
- процессный чек-лист.

## 3. Работа с релизами

### 3.1. Карточка просмотра релиза (view)

Это всплывающее окно поверх календаря:

- Заголовок: **Название релиза**.
- Поля:
    - Дата `dd.MM.yy`
    - Время `HH:mm`
    - Пак: *Название пака* + стрелка → переход к полной карточке пака
    - Платформы: чекбоксы (ТГ, Бусти, ВК, Tumblr)
- Иконки:
    - закрыть (X),
    - редактировать (карандаш),
    - удалить (корзина).

Открывается:

- либо по клику на карточку релиза в правой панели («Релизы»),
- либо по клику на релиз внутри клетки календаря.

Стикер: из карточки релиза можно перейти на полную карточку пака.

### 3.2. Карточка создания / редактирования релиза (edit)

Та же форма, но с инпутами:

- Название и время (поле названия релиза).
- Дата (с мини-календарём, по умолчанию — выбранный день в календаре).
- Время.
- Пак — выбор из выпадающего списка.
- Платформы — набор чекбоксов: ТГ, Бусти, ВК, Tumblr.

Открывается:

- по клику на пустой день календаря («тело дня») — создание релиза,
- по кнопке «Запланировать новый релиз» в правой панели,
- по кнопке «редактировать» из карточки просмотра.

### 3.3. Отображение релизов в календаре

- В ячейке дня отображается «релиз + статус» (например, иконкой или маленьким бейджем).
- При клике — всплывающее окно просмотра релиза.
- Общий вид: много дней с мелкими подписью «релиз + статус», справа — список паков.

## 4. Связки между сущностями

Из макетов очень чётко просматривается доменная модель:

- **Pack (Пак)**
    - Название RU / ENG
    - Тип, теги
    - Статус (workflow пака)
    - Файлы: обложка, скрины, дополнительные
    - Чек-лист задач
    - (Опционально) привязка к релизам
- **Release (Релиз)**
    - Название релиза
    - Дата + время
    - Ссылка на Pack
    - Набор платформ (TG / VK / Boosty / Tumblr)
    - Статус релиза (пока в макете просто «релиз + статус» внутри клетки, но поле явно подразумевается)
- **Platform**
    - В макете явно не выделена как отдельная сущность, но логика чекбоксов под это подталкивает.

По UX:

- Пак — создаётся и редактируется независимо от даты.
- Релиз — всегда привязан к:
    - конкретному дню/времени,
    - конкретному паку,
    - набору платформ.

Это хорошо совпадает с реальным процессом:

один пак может иметь несколько релизов (ранний доступ на Boosty, общий доступ в TG/VK и т.п.).

## 5. Что особенно удачно

1. **Чёткое разделение «Пак» vs «Релиз»**
    - Пак — про контент и файлы.
    - Релиз — про «когда и где публикуем».
2. **Календарный фокус**
    - Жена сразу видит, *что и когда* выходит, а не тонет в списках.
3. **Боковая панель «Следующие события»**
    - Удобный «мини-дашборд»: либо список паков, либо список релизов.
4. **Чек-листы внутри карточек пака**
    - Очень точно ложится на её реальный флоу («сделать позы → протестировать → скрины → обложка → посты»).
5. **Связка релиз → пак**
    - Возможность одним кликом из релиза попасть в полный пак даёт понятный навигационный мостик.

## 6. Места, где видно потенциальное развитие

Я не критикую, а отмечаю, что макеты логично расширятся дальше:

1. **Менеджер постов как отдельная сущность**
    - В чек-листе сейчас есть «подготовить пост для Telegram/VK/Boosty/Tumblr», но сами тексты/хэштеги/картинки постов нигде не хранятся.
    - По контенту, который ты показывал (посты из ТГ/VK/Boosty), напрашивается:
        - отдельный блок в паке/релизе: «Черновики постов по площадкам»
            
            (текст, хэштеги, какие картинки брать из файлов пака и т.п.).
            
2. **Статусы релиза и платформ**
    - В календаре видно «релиз + статус», но пока не расшифровано:
        - `planned / ready / posted`
        - отдельно по каждой платформе (например: Boosty — posted, VK — planned).
    - Это чуть позже можно будет формализовать, чтобы календарь показывал не просто «есть релиз», а «что уже опубликовано, а что ещё нет».
3. **Где показывать превью картинок**
    - В полной карточке пака это есть,
    - В правой панели (краткие карточки паков/релизов) можно будет добавить маленькую обложку — визуально станет ближе к реальным постам.
4. **Роли/юзеры**
    - Сейчас макеты под одного автора. Если потом добавишь, например, «комментарии» или «исполнителя задачи» — потребуется чуть расширить модель пользователя. Пока можно оставить как есть.

---

В целом макеты уже очень хорошо отражают реальный рабочий процесс:

- **Создаём пак** → наполняем его файлами и чек-листом.
- **Планируем один или несколько релизов** этого пака в календаре, с указанием платформ.
- **Смотрим месяц** и видим, когда какой контент выходит и на какие площадки.
